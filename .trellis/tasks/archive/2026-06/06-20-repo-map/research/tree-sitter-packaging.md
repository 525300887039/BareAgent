# Research: Python tree-sitter packaging ecosystem (2026) for Repo Map symbol skeletons

- **Query**: 2026 Python tree-sitter packaging for a "repo map" feature extracting class/function signature skeletons (Python + JS/TS + Rust min), new deps in an optional extra, fail-open, Windows + Python 3.12+.
- **Scope**: external (PyPI/GitHub live metadata, fetched 2026-06-20)
- **Date**: 2026-06-20

## TL;DR

- **`tree-sitter` (core bindings)**: latest **0.25.2** (2025-09-25), MIT, `requires-python >=3.10`, prebuilt **Windows `win_amd64` + `win_arm64` wheels for cp310–cp314** (incl. 3.12/3.13). API stabilized; the 0.21→0.22 break is long behind us.
- **`tree-sitter-language-pack`**: latest **1.9.1** (2026-06-18), MIT, **306+ languages**, ships **abi3 wheels cp310+ (one wheel covers 3.10–3.14)** including `win_amd64`/`win_arm64`. Actively maintained (repo pushed 2026-06-19). The wheel ships **only the compiled grammars + Python/Rust API — ZERO standalone `.scm` files on disk**. BUT its 1.9.x API now exposes **`get_tags_query(name) -> str | None`** ("if bundled") plus `get_highlights_query`/`get_locals_query`/`get_injections_query` — i.e. tag queries may be embedded in the native binary (coverage per-language **undocumented, must verify at runtime**).
- **`grep-ast`** (Aider's helper): latest **0.9.0** (2025-05-08), Apache-2.0, deps = `tree-sitter-language-pack` + `pathspec`. **Does NOT ship any `tags.scm`.** It provides `filename_to_lang()` (extension→language map for ~150 exts) + `TreeContext` (AST-aware grep rendering) + a thin `get_language`/`get_parser` re-export with fallback to old `tree-sitter-languages`. **Not the source of repo-map queries.**
- **Tag queries (`.scm`)**: Aider vendors its own at **`aider/queries/tree-sitter-language-pack/*-tags.scm`** (32 files incl. python/javascript/rust/go/java/c/cpp/csharp/ruby), Aider repo is **Apache-2.0** — freely vendorable.
- **`tree-sitter-languages`** (the OLD pack): **stale**, last release **1.10.2 (2024-02-04)**, Apache-2.0. Superseded by `tree-sitter-language-pack`. Do not use for new code.

## Findings

### 1. Package options & current state (2026)

#### `tree-sitter` (core Python bindings)
| Field | Value |
|---|---|
| Latest | **0.25.2** (2025-09-25) |
| License | MIT |
| requires-python | `>=3.10` |
| Windows wheels | yes — `win_amd64` **and** `win_arm64`, for cp310/311/312/313/314 |
| Other wheels | macOS x86_64+arm64, manylinux 2.17/2.28 aarch64+x86_64, musllinux |

API stability: the disruptive break was **0.21 → 0.22** (Language/Parser construction changed). As of 0.22+ (and stable through 0.25.x) the idiom is:
```python
from tree_sitter import Language, Parser, Query
PY_LANGUAGE = Language(tspython.language())   # wrap the PyCapsule
parser = Parser(PY_LANGUAGE)                  # language passed to constructor (no more parser.set_language)
tree = parser.parse(bytes(src, "utf8"))
query = Query(PY_LANGUAGE, scm_text)          # Query(language, query_string)
```
Release cadence 0.22 (2024-05) → 0.23 (2024-08) → 0.24 (2025-01) → 0.25 (2025-07) — slowing, no churny breaks recently. **Note**: when going through `tree-sitter-language-pack`, you almost never touch the raw `tree-sitter` API for construction — TSL pack returns ready `Language`/`Parser` objects, insulating you from this. But you still need `tree_sitter.Query` to run `.scm` queries against a tree (TSL pack re-exports `Language`/`Parser`/`Node`/`Tree`/`Query`-adjacent types from its own `_native`).

#### `tree-sitter-language-pack`
| Field | Value |
|---|---|
| Latest | **1.9.1** (2026-06-18); 1.9.0 same day; 1.8.x in May 2026 |
| License | **MIT** (`License-Expression: MIT`, "Copyright 2025–2026 Na'aman Hirschfeld") |
| Summary | "Pre-compiled tree-sitter grammars for 306 programming languages" |
| requires-python | `>=3.10`; classifiers list 3.10–3.14 |
| Wheels | **abi3** (`cp310-abi3-*`) — a single wheel works on 3.10 through 3.14. Platforms: `win_amd64`, `win_arm64`, macOS x86_64+arm64, manylinux_2_34 aarch64+x86_64. |
| Repo | `kreuzberg-dev/tree-sitter-language-pack` (formerly `Goldziher/...`, 301-redirects), pushed **2026-06-19**, ~393 stars. Actively maintained. |
| Transitive deps | none declared (`requires_dist: None`) — grammars are compiled into the wheel. |

**Architecture note (2026 shift)**: 1.9.x is a **Rust-native binding** — the wheel contains `tree_sitter_language_pack/_native.pyd` (auto-generated by a tool called `alef`). API is in `api.py` + `__init__.py`. Public surface includes:
- `get_language(name) -> Language`, `get_parser(name) -> Parser` (these are what grep-ast imports — still present, so grep-ast stays compatible).
- `get_tags_query(name) -> str | None`, `get_highlights_query`, `get_locals_query`, `get_injections_query`.
- `detect_language(path)`, `available_languages()`, `has_language(name)`, `language_count()`, plus a lazy `download*` family (on-demand grammar download for languages not in the base wheel).

**Does it ship `.scm` files?** Not as files on disk (the wheel has only `_native.pyd` + a handful of `.py`). It exposes them via **`get_tags_query(language)`** which returns the query string "**if bundled**" (else `None`). So the queries are embedded in the native binary for some languages — but **which languages have a bundled tags query is undocumented** and the README does not describe coverage. **This must be probed at runtime** (e.g. `get_tags_query("python")`, `"javascript"`, `"typescript"`, `"rust"`) before relying on it. Treat as a nice-to-have fast path, not a guarantee.

#### `grep-ast` (Aider's helper)
| Field | Value |
|---|---|
| Latest | **0.9.0** (2025-05-08); 0.8.x March 2025 |
| License | **Apache-2.0** |
| requires-python | unset (`None`) |
| Wheel | `grep_ast-0.9.0-py3-none-any.whl` (pure Python — no compiler, works everywhere incl. Windows) |
| Deps | `tree-sitter-language-pack`, `pathspec` |

What's actually inside (verified by unzipping the wheel — only 6 modules): `grep_ast.py` (`TreeContext`), `parsers.py` (`PARSERS` extension→lang dict + `filename_to_lang()`), `tsl.py` (a 9-line shim: `from tree_sitter_language_pack import get_language, get_parser`, falling back to old `tree_sitter_languages`), `dump.py`, `main.py` (CLI entry), `__init__.py`. **No `.scm` files. No tags/symbol extraction logic.** grep-ast's job is "AST-aware grep with context" (the `TreeContext` renderer), NOT repo mapping.

**Implication**: depending on grep-ast gets you (a) `tree-sitter-language-pack` transitively (grammars + Windows wheels), (b) a maintained `filename_to_lang()` extension map, (c) `pathspec`. It does **not** get you the tag queries — those still come from Aider's vendored `.scm` files or from TSL pack's `get_tags_query`. So "depend on grep-ast" is **lower-effort for the plumbing, but you still supply the queries.**

#### `tree-sitter-languages` (the predecessor)
Last release **1.10.2 (2024-02-04)**, Apache-2.0, dep `tree-sitter`, lots of per-cpython wheels (cp37–cp312) incl. `win_amd64`. **Stale / superseded** by `tree-sitter-language-pack`. grep-ast keeps it only as an import fallback. Don't pick it for new work.

#### Other 2026 options
- **Per-language grammar packages** (`tree-sitter-python`, `tree-sitter-javascript`, `tree-sitter-rust`, `tree-sitter-typescript`, …): individual MIT wheels, each pins a `tree_sitter>=0.23.x` range, each ships its own grammar. You'd add one dep per language + still vendor queries. More deps to track than the language-pack; only worth it if you want a very small fixed language set and dislike a 300-language binary. (The core `tree-sitter` test extra pins `tree-sitter-python>=0.23.6`, `-javascript>=0.23.1`, `-rust>=0.23.2` — confirms current per-lang versions.)

### 2. Tag queries (`.scm`) — who ships them

- **`grep-ast`**: none.
- **`tree-sitter-language-pack` 1.9.x**: maybe, via `get_tags_query(name) -> str | None` (embedded in `_native`, coverage unverified — probe at runtime).
- **`tree-sitter` core / per-lang grammar packages**: the grammar wheels generally do **not** ship `tags.scm` (some ship `highlights.scm` only; tags is a Aider/ctags-style convention, not part of the upstream grammar).
- **Aider (canonical source for repo-map tags)**: `aider/queries/tree-sitter-language-pack/*-tags.scm`. Confirmed 32 files via GitHub API:
  `arduino, bash, c, chatito, clojure, commonlisp, cpp, csharp, d, dart, elisp, elixir, elm, gleam, go, java, javascript, lua, matlab, ocaml, ocaml_interface, pony, properties, python, r, racket, ruby, rust, solidity, swift, udev` (+ `README.md`). There's also a parallel `aider/queries/tree-sitter-languages/` dir for the old pack.
  - **TypeScript caveat**: there is **no `typescript-tags.scm`** in the list above. Aider/grep-ast map `.ts`/`.tsx` → language name `"typescript"`; Aider's actual TS coverage historically reuses/derives queries. If we want TS we likely vendor/author a `typescript-tags.scm` (TS grammar nodes differ from JS; the JS query won't fully cover TS). Verify against the live Aider tree at implementation time.

Example of what a vendored query looks like (these are tiny, ~5–40 lines each):
```scheme
; python-tags.scm (Aider, full file)
(class_definition name: (identifier) @name.definition.class) @definition.class
(function_definition name: (identifier) @name.definition.function) @definition.function
(call function: [ (identifier) @name.reference.call
                  (attribute attribute: (identifier) @name.reference.call) ]) @reference.call
; (+ module-level constant assignment)
```
The `@definition.class` / `@definition.function` / `@name.definition.*` capture naming convention is the de-facto standard (ctags-derived). To get **signature skeletons** (decl line, not body) you take each `@definition.*` node's start line through the end of its signature (e.g. up to the `:` for Python, the `{`/`;` for C-likes) rather than the node's full end — that's repo-map logic on top of the captures, not something the `.scm` gives you directly.

### 3. Windows + Python 3.12 wheel availability (critical — user on win32)

All viable packages ship **prebuilt Windows wheels — no C compiler needed at install**:
- `tree-sitter` 0.25.2: `cp312-cp312-win_amd64.whl` (and win_arm64), also cp310–314.
- `tree-sitter-language-pack` 1.9.1: `cp310-abi3-win_amd64.whl` (one abi3 wheel covers 3.12 + 3.13 + 3.14) and `win_arm64`.
- `grep-ast` 0.9.0: `py3-none-any.whl` (pure Python, trivially installs on Windows; its only compiled dep is TSL pack, which has the Windows wheel above).
- Local interpreter here is **Python 3.14.4** — covered by all the above (tree-sitter has cp314 wheels; TSL pack abi3 covers it; grep-ast is pure-py). Project floor is 3.12, also fully covered.

No source build / MSVC toolchain required on Windows for any of the recommended sets.

### 5. Recommendation (minimal-footprint, ranked)

Goal: multi-language class/function **signature** extraction (Python + JS/TS + Rust min), one optional extra, fail-open when not installed.

**Ranked options:**

1. **(b-lite, RECOMMENDED) `tree-sitter` + `tree-sitter-language-pack`, vendor a small set of `.scm` tag queries.**
   - Extra: `repo-map = ["tree-sitter>=0.25", "tree-sitter-language-pack>=1.9"]` (mirrors existing `lsp`/`pdf`/`embeddings` extras).
   - Vendor ~4–8 `*-tags.scm` from Aider (python, javascript, rust, go, java, c, cpp, ruby — pick the subset matching ambitions; min = python + javascript + rust; author/borrow `typescript-tags.scm` if TS wanted) into e.g. `src/bareagent/memory/repo_map_queries/`. Apache-2.0 attribution.
   - Code path: `from tree_sitter_language_pack import get_parser, get_language` → parse file → `tree_sitter.Query(lang, scm_text)` → walk `@definition.*` captures → slice the signature line(s).
   - **Why #1**: full control over exactly which captures/languages, no surprise query coverage, queries are static text we own and can test deterministically (fits the "纯逻辑模块 + 注入式可单测 + fail-open" pattern from `code_index.py`). Windows wheels guaranteed. Avoids depending on grep-ast's `TreeContext` (which we don't use) and pathspec (we already have `iter_search_files`).
   - **Cost**: we write the capture→signature slicing logic + carry the `.scm` files (small, stable).

2. **(b-fast-path) Same deps as #1, but try `tree_sitter_language_pack.get_tags_query(lang)` first, fall back to vendored `.scm`.**
   - Lets you skip vendoring for whatever languages TSL pack bundles, while staying correct for the rest.
   - **Caveat**: per-language bundled-tags coverage is undocumented and returns `None` when absent — so you still need the vendored fallback for guaranteed Python/JS/TS/Rust. Net: same vendoring effort as #1 for the must-have langs, with an optional shortcut for extras. Only marginally better than #1; adds a runtime branch. Worth it only if you want broad opportunistic language coverage cheaply.

3. **(a, lowest plumbing effort) Depend on `grep-ast`, still vendor `.scm`.**
   - Extra: `repo-map = ["grep-ast>=0.9"]` (pulls TSL pack + pathspec transitively).
   - Get `filename_to_lang()` extension map for free (~150 extensions, maintained).
   - **But**: grep-ast ships no queries, so you still vendor the `.scm` files exactly as in #1. You gain a maintained ext→lang map and lose nothing, but pull an extra (`pathspec`) and a CLI we won't use. Since we already have `iter_search_files` (`core/handlers/search_utils.py`) and only need ~3–8 extension mappings (a 10-line dict), grep-ast's main value-add is marginal for our scoped language set.
   - Pick this only if we want grep-ast's broad extension map and don't mind the extra transitive dep.

**Bottom line**: there is **no "depend on one package and get queries for free" path** in 2026 — every route requires us to either vendor Aider's Apache-2.0 `.scm` files or rely on TSL pack's undocumented `get_tags_query`. Given the project's existing pattern (small, owned, testable pure-logic modules), **Option 1** is the cleanest: `tree-sitter-language-pack` for grammars+Windows wheels, `tree-sitter` for the `Query` runtime, and a handful of vendored Apache-2.0 `.scm` files we control. Adding the `get_tags_query` fast-path (Option 2) is a low-risk future enhancement once the base path works.

## Caveats / Not Found

- **`get_tags_query` per-language coverage unverified**: the wheel only contains a compiled `_native.pyd`, so I could not enumerate which languages have bundled tags queries. The API contract is "returns the string if bundled, else None". Must be probed at runtime against the target version (`get_tags_query("python"/"javascript"/"typescript"/"rust")`) before relying on it for any language.
- **TypeScript tags query**: not present as `typescript-tags.scm` in Aider's `tree-sitter-language-pack` queries dir at fetch time (only `javascript-tags.scm`). Confirm Aider's current TS handling / author a TS query if TS is in scope. (TSX/TS use distinct grammars from JS; JS query is insufficient for full TS coverage.)
- **Signature-only extraction is repo-map logic, not query output**: the `.scm` captures give you *definition nodes*; turning a node into a one-line signature skeleton (and stripping the body) is code we write. Aider does this in its `repomap.py`; that file is Apache-2.0 if we want to reference the approach (PageRank ranking is a separate, out-of-MVP concern per the PRD).
- Exa MCP tools were unavailable in this session; all facts above come from **live PyPI JSON, downloaded/unzipped wheels, and the GitHub API/raw** (fetched 2026-06-20), which is more authoritative than search snippets for versions/wheels/file listings.
